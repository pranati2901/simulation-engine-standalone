SimCore — EV Data Import Spec

The exact data we need, and the format our platform ingests

NextXR · SimCore
How it works. SimCore never invents results — it builds a live model of your network from the data below, and the simulation engine computes every outcome on your assets, tariffs and topology. This document is the import contract: a network provided in the format on the right loads straight into the platform's Data Layer and drives the twin, the fault simulations and the AI copilot. A pilot can start with a SINGLE SITE and expand.

2 · The import format this is what the platform reads

Provide the network as one JSON object (or a CSV we map into it). This exact shape loads via the Data Layer's Import network:

// one site — paste into Data → Import network, or send us the file
{
  "site": "Mall Plaza · City Centre",
  "currency": "INR",
  "cost": {
    "rev_inr_per_kwh": 18,           // price to the driver
    "penalty_inr_per_hour_down": 1200,  // SLA penalty per hour a station is down
    "demand_charge_inr_per_kw": 350     // utility demand charge
  },
  "assets": [
    { "id": "TX-1",   "name": "Transformer T1", "type": "transformer",  "rating_kw": 630,  "faults": ["overload", "overheat"] },
    { "id": "F-1",    "name": "Feeder F-1",     "type": "feeder",       "capacity_kw": 400, "faults": ["overcurrent_trip"] },
    { "id": "DCFC",  "name": "DC-fast bank",   "type": "charger_bank", "chargers": 8, "kw": 120, "faults": ["charger_offline", "connector_fault"] },
    { "id": "BESS-A", "name": "BESS-A",        "type": "bess",   "capacity_kwh": 200, "power_kw": 100, "faults": ["thermal_runaway", "offline"] },
    { "id": "GRID",  "name": "Grid supply",   "type": "grid",   "faults": ["brownout", "supply_loss"] }
  ]
}

3 · Field reference

FieldRequiredNotes
siteyesDisplay name of the site/network
cost.rev_inr_per_kwh / penalty_inr_per_hour_down / demand_charge_inr_per_kwyesYour tariff & SLA — every ₹ figure derives from these
assets[].id / nameyesStable code + label (e.g. TX-1, "Transformer T1")
assets[].typeyesOne of transformer · feeder · charger_bank · bess · grid
assets[].faultsyesFault ids valid for the type — see below
rating_kw / capacity_kw / chargers / kw / capacity_kwh / power_kwoptionalRatings that sharpen the model per asset type

Valid fault ids by type:

  • transformer → overload, overheat  ·  feeder → overcurrent_trip  ·  charger_bank → charger_offline, connector_fault
  • bess → thermal_runaway, offline  ·  grid → brownout, supply_loss

4 · What changes when you provide data — and what doesn't (honest)

  • Becomes yours: the asset list, the fault menu, every ₹ figure (your tariffs & SLA), the AI copilot's grounded answers, the Monte-Carlo probabilities, and the mitigation-strategy ranking.
  • Stays representative: the 3-D scene remains a stylised charging-hub twin — it doesn't auto-generate bespoke geometry from your layout. Numbers and reasoning are yours; the 3-D is illustrative.
  • Deployment adds: live OCPP/SCADA streaming (this doc is the one-time export; a pilot then wires the live feed).

Format, scope & security

  • Format: the JSON above, a CSV we map to it, or a read-only API. Standard OCPP — no changes to your stack.
  • Minimum to start: site, cost, and a few assets for one site. Ratings & history are optional and improve accuracy.
  • Security: read-only; anonymisable (asset codes, not addresses); no control signals are ever sent to your equipment.
